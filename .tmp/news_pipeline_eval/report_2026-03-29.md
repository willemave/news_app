## Observed Findings

### 1. The digest candidate cap is active in all three runs

- Every run started with `200` input items but the persisted digest only reports `source_count=150`.
- This matches the current setting `news_digest_max_candidates = 150`.
- Result: the harness processed all `200` items, but digest curation only saw the top `150`.

### 2. The synthetic run exercised real summary generation

- The synthetic corpus had `40` items without an explicit `metadata.summary`.
- The pipeline generated new summaries for `30` of them and reused existing summaries for `170`.
- The missing `10` came from title-only items where ingestion populated `summary_title` from `article_title`, so `process_news_item` treated them as already summarized.
- This is real behavior, not an inference: the run artifact reports `generated_summary_count=30`.

### 3. The synthetic case over-collapsed into very large curated bullets

- Final output was only `3` bullets from `150` digest candidates.
- Those bullets cited `32`, `20`, and `12` items respectively.
- This indicates the synthetic corpus is useful for stress-testing batch curation and cost, but it is too template-homogeneous to be a strong clustering-quality benchmark.
- It effectively encouraged the model to roll many similar items into a handful of umbrella stories.

### 4. The production snapshot runs used real production data, but they did not trigger fresh summarization

- Both production runs processed `200` items and generated `0` new summaries.
- They reused summaries for all `200` items.
- This is because the exported production `news_item_records` already contain enough summary state to satisfy `process_news_item`.
- In the sampled production inputs:
  - Case A had `65` title-only rows and `135` rows with summary text or key points.
  - Case B had `66` title-only rows and `134` rows with summary text or key points.
- Conclusion: these snapshot runs validate real production evidence flowing through processing, clustering, and digest generation, but they do not validate raw-scrape-to-summary regeneration.

### 5. Production curation is extremely selective right now

- Case A cited only `9` unique items out of `150` digest candidates.
- Case B cited only `10` unique items out of `150` digest candidates.
- That is only `6.0%` and `6.7%` candidate coverage respectively.
- The final digests were coherent, but they were very narrow relative to the candidate pool.

### 6. The mixed user-visible production sample still produced a mostly global digest

- Case B contained `19` user-scoped items and `181` global items.
- The final digest cited `0` user-scoped items and `10` global items.
- In this random sample, the user-private X items did not survive curation.
- This suggests current curation is dominated by global news importance rather than user-specific relevance.

### 7. Noise suppression is working better on production data than on synthetic data

- Case A’s sampled production input clearly included low-value rows such as blocked Reddit pages and `Untitled` items.
- Those low-signal rows did not surface in the final bullet set.
- The final production bullets focused on topics like:
  - Moonshot / Kimi integration and AttnRes
  - Meta Reality Labs strategy
  - Together AI funding
  - Floci / Local AWS emulation
  - OpenAI funding
  - Disney/OpenAI partnership collapse
  - Trivy supply-chain compromise

### 8. Latency is dominated by summary generation, not BAAI embeddings

- Synthetic run: `59s`
- Production snapshot runs: `11s` each
- The largest structural difference is that the synthetic run performed `30` real summary generations while the production snapshot runs performed `0`.
- This supports the earlier hypothesis that the expensive part is still the LLM path, not BAAI.

## Practical Conclusions

- The new harness is working for both synthetic and production-derived cases.
- The snapshot path is already useful for real-world digest quality checks against `news_app.db`.
- If the goal is to test true raw-content summarization on production-shaped data, the current snapshot exporter is not enough by itself because it preserves summary-ready state.
- The current digest is probably too selective for random 150-item production candidate sets.
- The current user-preference path is not yet strong enough to pull user-scoped X items into the final digest when mixed with broad global news.

## Recommended Next Checks

- Add a second exporter mode that strips summary fields from a controlled subset of production rows, so snapshot evals can intentionally force fresh summarization.
- Add a report field for raw cluster count before curation, so over-selection vs over-clustering is visible without digging into DB internals.
- Add a recall-oriented metric to the report: cited unique items divided by digest candidates, because the current artifacts show strong precision but very low coverage on random production samples.
